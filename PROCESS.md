# Process — adding a book to the library

This is the canonical, end-to-end guide for taking a book and turning it into a
fully-integrated entry in the Skills from Books library. It covers both the
content side (extracting skills, writing the loglog) and the infrastructure
side (building JSON, registering the book on the parent index, wiring up the
viewer pages). Every decision the existing books locked in is documented here.

If you only want a one-off skill from a short article or conversation, write
the `SKILL.md` directly using the format in section 4 below — you can skip the
book-level scaffolding.

---

## 0. Inputs and outputs

Inputs you'll need before starting:

- The book itself (epub, pdf, or txt — epub is preferred because chapter
  boundaries are explicit).
- A short slug for the book: lowercase, dash-separated, one or two words.
  Examples: `saunders`, `feedback`, `difficult`, `polysecure`.
- A category: pick from the existing categories (`Literature`, `Communication`,
  `Relationships`) or add a new one. Categories live in
  `docs/index.html` as `<div class="category">` blocks.

Outputs you'll have when done:

- `skills/<slug>/_book.md` — the parent context every skill in the book inherits
- `skills/<slug>/<skill-slug>/SKILL.md` — atomic skills, one folder each
- `skills/<slug>/recipes/<recipe-slug>/SKILL.md` — composed recipes (this book)
- `skills/cross-book/<recipe-slug>/SKILL.md` — composed recipes that cross
  books (only if the recipe genuinely needs atomic skills from multiple books)
- `docs/data/<slug>.loglog` — the conceptually-ordered reading log
- `docs/data/<slug>-skills.json` — the built JSON for the skills viewer
  (generated by `docs/build.py`, do not hand-edit)
- `docs/<slug>/index.html` — the skills viewer page for this book
- `docs/<slug>/log/index.html` — the reading-log viewer page for this book
- `docs/index.html` — updated with a new card for this book in its category

## 1. Setup (one-time per book)

### 1a. Get the source text

Extract the source to plain text. Don't read the epub or pdf directly during
extraction — layout artifacts will distort line-by-line work.

For epubs, the simplest path is a small Python extractor:

```python
import zipfile, re
from html.parser import HTMLParser

class T(HTMLParser):
    def __init__(self):
        super().__init__(); self.parts=[]; self.skip=False
    def handle_starttag(self, tag, attrs):
        if tag in ('script','style'): self.skip=True
        elif tag in ('p','div','h1','h2','h3','h4'): self.parts.append('\n\n')
        elif tag=='li': self.parts.append('\n- ')
        elif tag=='br': self.parts.append('\n')
    def handle_endtag(self, tag):
        if tag in ('script','style'): self.skip=False
    def handle_data(self, data):
        if not self.skip: self.parts.append(data)

with zipfile.ZipFile('book.epub') as z:
    parts = []
    for n in sorted(n for n in z.namelist() if n.endswith('.html')):
        with z.open(n) as f:
            ex = T(); ex.feed(f.read().decode('utf-8','replace'))
            parts.append(re.sub(r'\n{3,}','\n\n', ''.join(ex.parts)).strip())
    open('book.txt','w').write('\n\n'.join(parts))
```

For pdfs, use `pdftotext` (`poppler-utils`). For .mobi, use `ebook-convert`
(`calibre`).

Save the extracted text alongside the source.

### 1b. Decide the category

Look at `docs/index.html` for the existing categories and pick the one that
fits. If none does, add a new `<div class="category">` block with a new
`<div class="category-label">` heading. The categories so far:

- **Literature** — books about reading, writing, or interpreting works
- **Communication** — books about how to talk and listen with others
- **Relationships** — books about attachment, intimacy, partnership

### 1c. Create directories

```
skills/<slug>/
skills/<slug>/_process/
skills/<slug>/recipes/
docs/<slug>/
docs/<slug>/log/
```

`_process/` and `recipes/` get a `.gitkeep` until they have content.

### 1d. Write `skills/<slug>/_book.md`

Skim the table of contents, preface, and introduction. Capture in the book
header:

```markdown
---
book: <slug>
title: <full title>
subtitle: <if present>
authors:
  - First Last
year: 2020
publisher: ...
domain: relationships  # one of: literature, communication, relationships, ...
description: <one-paragraph framing>
---

<2-3 paragraph orientation: what the book is, what its lens is, what
vocabulary recurs, what the reader-of-this-skills-library should keep in
mind when reading skills extracted from it>
```

This file is parent context every skill inherits, which lets individual
skills stay small.

## 2. Write the loglog file

Path: `docs/data/<slug>.loglog`

The loglog file is the **conceptually-organized** distillation of the book.
It is **not** a chapter-by-chapter outline — readers of the loglog viewer
should be able to find a concept without knowing which chapter it lived in.

### 2a. Format

Indent-based tree, two-space increments. Each top-level bullet is a major
concept; children are sub-concepts; leaves are sentence-length statements.
A typical book lands at 200–500 lines.

```
- Top-level concept
    - Sub-concept
        - A specific point
        - Another specific point
        - The point with examples and the example itself: "..."
    - Another sub-concept
        - ...
- Next top-level concept
    - ...
```

### 2b. What goes in

- Every major framework, named technique, taxonomy
- Specific chapter examples that make the abstraction concrete
- Connecting tissue: "the X relates to Y because..."
- Notable quotes and definitions, in quotes, attributed inline ("X (Author)")

### 2c. What stays out

- Page-by-page summary
- Chapter-recap sentences ("In chapter 3, the author discusses...")
- Long verbatim passages (use brief quotes when the exact phrasing is the
  technique)

### 2d. Conceptual hierarchy, not chapter order

The loglog of `polysecure.loglog` is a model: it goes attachment basics →
the four styles → nested model → nonmonogamy → HEARTS → secure self →
boundaries → final thoughts. The book's chapters are reordered to make the
ideas build on each other rather than to mirror reading order.

## 3. Per-chapter extraction pass for skills

This is the content work. The web build, registration, and tags happen later.

### 3a. The candidate gate

A candidate becomes a skill if **all four** of these are true:

1. You can write a concrete action (the move itself, not a description of it)
2. You can write an observable signal that tells you it worked
3. You can write a counter — the recognizable failure mode the skill prevents
4. The move is not already obtainable from an existing skill in the library

If any are false, the candidate is dropped or folded into an existing skill.

### 3b. Per chapter

For each chapter:

1. Read end-to-end without extracting (hold the shape in mind)
2. Re-read line by line. Append candidates to
   `skills/<slug>/_process/candidates.log`:
   ```
   - Candidate: <one-line paraphrase>
       location: ch3, page 47
       situation: <one line>
   ```
3. Run the gate on each candidate.
4. For passing candidates, create `skills/<slug>/<skill-slug>/SKILL.md`
   (format in section 4).
5. For failing candidates, fold into an existing skill or drop.
6. Mark each candidate with its outcome in the log:
   `→ promoted to <slug>`, `→ folded into <slug>`, `→ dropped because <reason>`.

### 3c. Saturation rule

Each chapter terminates explicitly:

1. **Extract pass** — section 3b.
2. **Saturation check** — append to `skills/<slug>/_process/saturation.log`:
   - kinds of moves that came out
   - kinds of moves the chapter is about that didn't pass the gate
   - why those didn't make it
3. **No squeezing.** Once saturated, the chapter is closed for new skills.
   Re-reads are for refactoring only. If a re-read genuinely surfaces something
   new, the saturation note was wrong — record the reopen with the new
   candidate and update the note.
4. **Refactor pass is its own activity.** Each pass logs to
   `skills/<slug>/_process/refactor.log`.

## 4. SKILL.md format

The canonical structure. Every skill file has this shape.

### 4a. Frontmatter

```yaml
---
name: <skill-slug>          # match the folder name
description: <one paragraph naming the recognizable situation in which
  the skill should fire — this is what skill-loaders match against>
type: atomic                 # or "composed" for recipes
book: <slug>                 # the book slug
sources:
  - chapter: "Chapter Three"
    lines: "188-278"         # optional, line range in the source text
    note: "<short framing of what this section establishes>"
tags:
  - <canonical tag>
  - <canonical tag>
related:
  - extends: <slug-of-other-skill>      # optional
  - refines: <slug-of-other-skill>      # optional
  - contrasts: <slug-of-other-skill>    # optional
---
```

Cross-book references in `related:` use the bare skill slug. The viewer's
linkifier resolves them within the current book; references to skills in
other books can use `<other-book>/<skill-slug>` form (e.g.
`feedback/take-responsibility-first-then-introduce-the-system`) and the
viewer strips the prefix when matching.

### 4b. Body sections (in this order)

Required for atomic skills:

```markdown
## When to reach for this

<paragraph: the recognizable situation, written so the reader can match
their own current situation against it>

## The move

1. <step 1, complete in itself>
2. <step 2>
...

## Signal it landed

<paragraph: what's observably true if the skill worked>

## Probe before & after

<a simple check the practitioner can run on themselves before and after>

## Failure mode this prevents

<paragraph: what goes wrong if the skill is skipped>

## Worked micro-example

<short scenario the source book actually uses, paraphrased>

## Reflection prompts            (OPTIONAL — see section 4c)

- <question 1>
- <question 2>

## Source

<chapter and section in the source book; what concept the section establishes>

## Related skills

- <type> `<other-skill>` — <one-line condition for reaching for it instead>
```

The viewer renders sections in this exact order, with these exact labels.
The build script in `docs/build.py` maps `## Header → key` via
`SECTION_KEYS`; the viewer label map in
`docs/shared/skills-viewer.js` (`SECTION_LABELS`) must agree.

### 4c. Reflection prompts (for books that have end-of-chapter questions)

Some books (Polysecure being the canonical example) include end-of-chapter
"Questions to Consider" / "Reflection Questions". When a skill maps to a
chapter that has these, include them as a `## Reflection prompts` section.

This section is rendered between `Worked micro-example` and `Source`. Each
prompt is its own bullet:

```markdown
## Reflection prompts

- <verbatim or near-verbatim from the book's end-of-chapter questions>
- <next question>
```

Books without end-of-chapter questions simply omit this section.

### 4d. Composed recipes

For composed skills (recipes), use `type: composed`. The `## The move`
section names the atomic skills that compose the recipe in order, with the
sequencing logic. See `skills/saunders/recipes/workshop-critique-of-a-draft/`
or `skills/polysecure/recipes/become-polysecure-with-self/` for examples.

### 4e. No invented examples

`Worked micro-example` paraphrases a scenario the book itself uses. If the
book has no concrete factual example to draw on, omit the section rather
than fabricate one.

### 4f. No extended verbatim text

Frameworks, named techniques, chapter pointers, and brief one-line attributed
quotes (when the exact phrasing IS the technique) are fair to cite. Long
passages are not.

## 5. Canonical tag vocabulary

After writing skills with whatever ad-hoc tags feel right, normalize against
the canonical set in `skills/CANONICAL_TAGS.md`. There are 40 canonical tags
grouped into 6 facets:

- **Phase**: `preparation`, `in-the-moment`, `revision`, `closing`
- **Action**: `diagnose`, `inquire`, `listen`, `reframe`, `assert`,
  `separate`, `sort`, `contain`, `experiment`, `cut`, `iterate`
- **Focus**: `self`, `text`, `other`, `exchange`, `relationship`
- **Topic**: `emotion`, `identity`, `intent-impact`, `contribution`,
  `triggers`, `purpose`, `distortion`, `coaching`, `evaluation`,
  `blind-spot`, `boundaries`, `accountability`
- **Craft** (story-craft only — saunders): `structure`, `voice`, `pacing`,
  `point-of-view`, `characterization`, `climax`, `endings`
- **Stance**: `curiosity`, `growth`

To normalize tags after writing skills:

```bash
cd skills
python3 update_tags.py
```

This script lives at `skills/update_tags.py`. Add the new book slug to the
`books` list in the script's `main()` if it isn't already there. The script
maps legacy tags through `TAG_MAP` and adds bonus tags inferred from skill
slugs via `SLUG_BONUS`.

If the book is in a non-craft domain (e.g. communication, relationships),
its skills should not pick up `structure`/`voice`/`pacing` etc. via the
slug-bonus regex (those metaphors will fire on words like "internal voice"
in feedback). Run `skills/strip_craft_tags.py` after `update_tags.py` if
needed.

The viewer's facet sidebar hides tags with zero matches in the current book,
so you don't need per-book FACETS configuration — just keep the data clean.

## 6. Build the JSON

```bash
cd docs
python3 build.py
```

`docs/build.py` walks `skills/<slug>/` (including `recipes/`) and emits
`docs/data/<slug>-skills.json`. **Add the new book slug to `BOOKS` in the
script** if it isn't already there.

The JSON is consumed by the skills viewer. Do not hand-edit it; rerun
`build.py` after any SKILL.md change.

## 7. Web pages: register the book

### 7a. Skills viewer page

Copy `docs/saunders/index.html` to `docs/<slug>/index.html` and replace
the four occurrences:

- `<title>Saunders — Skills</title>` → `<title>Polysecure — Skills</title>`
- `Saunders — A Swim in a Pond in the Rain` → `<your book label>` (twice:
  once in the page header span, once in `createSkillsViewer({title: ...})`)
- `dataFile: '../data/saunders-skills.json'` → `'../data/<slug>-skills.json'`
- `bookSlug: 'saunders'` → `'<slug>'`

### 7b. Reading-log viewer page

Copy `docs/saunders/log/index.html` to `docs/<slug>/log/index.html` and
replace the analogous strings, pointing `dataFile` at
`'../../data/<slug>.loglog'`.

### 7c. Parent index card

In `docs/index.html`, find the right `<div class="category">` block and
add a new `<div class="book-card">`:

```html
<div class="book-card">
  <div class="book-card-body">
    <div class="book-title">Polysecure</div>
    <div class="book-authors">Jessica Fern</div>
    <div class="book-desc">One-paragraph blurb of what this book contributes.</div>
  </div>
  <div class="book-card-links">
    <a href="<slug>/log/">Reading Log</a>
    <a href="<slug>/">Skills</a>
  </div>
</div>
```

### 7d. Cache-busting

When you update `skills-viewer.js`, `skills-viewer.css`, `loglog-viewer.js`,
or `loglog-viewer.css`, bump the `?v=...` query string on every HTML page's
script/link tags so browsers fetch the new version. The convention is a
date-stamp like `v=20260508` with a letter suffix for same-day bumps:
`v=20260508a`, `b`, `c`...

A small Python one-liner:

```python
import re
v = 'v=20260508a'
for p in [
    'docs/index.html',
    'docs/saunders/index.html', 'docs/feedback/index.html',
    'docs/difficult/index.html', 'docs/polysecure/index.html',
    'docs/saunders/log/index.html', 'docs/feedback/log/index.html',
    'docs/difficult/log/index.html', 'docs/polysecure/log/index.html',
]:
    t = open(p).read()
    t = re.sub(r'(skills-viewer\.js)(\?v=[^"]*)?', f'\\1?{v}', t)
    t = re.sub(r'(skills-viewer\.css)(\?v=[^"]*)?', f'\\1?{v}', t)
    t = re.sub(r'(loglog-viewer\.js)(\?v=[^"]*)?', f'\\1?{v}', t)
    t = re.sub(r'(loglog-viewer\.css)(\?v=[^"]*)?', f'\\1?{v}', t)
    open(p, 'w').write(t)
```

## 8. Cross-book references and recipes

### 8a. Cross-book references in `Related skills`

A skill in book A can reference a skill in book B by writing
`<book-b>/<skill-slug>` in the Related skills body. Example from Polysecure's
`turn-toward-after-rupture`:

```markdown
- Cross-references `feedback/take-responsibility-first-then-introduce-the-system` —
  the same shape applies: own your part before raising the system question.
```

The viewer's `linkifySkillReferences` strips the `book/` prefix when matching
against the in-book slug set, so bare slugs and prefixed slugs both render as
clickable deep-links.

### 8b. Cross-book recipes

When a recipe legitimately combines atomic skills from more than one book,
put it in `skills/cross-book/<recipe-slug>/` rather than in any one book's
`recipes/`. Cross-book recipes:

- Have `type: composed`
- Reference atomic skills with `<book>/<skill-slug>` notation
- Are mentioned in `skills/CATALOG.md` under appropriate scenarios

The build pipeline doesn't create a separate JSON for `cross-book/`; cross-
book recipes are not currently surfaced in the per-book viewers. If/when
they are, add a viewer page for them.

## 9. Per-book finalization

After all chapters are saturated:

1. **Cross-reference pass** — go through every skill in the book and set
   `related:` links to other skills in the library, in this book and
   elsewhere. Use the taxonomy in section 10.
2. **Recipe pass** — identify scenarios where two or more atomic skills
   fire in a known order. Write composed recipes (in this book's `recipes/`
   if all atomic skills are from this book; in `skills/cross-book/` if not).
3. **Catalog update** — open `skills/CATALOG.md` and add the new skills
   under relevant scenarios. Add new scenarios if the book introduced
   situations the catalog didn't have.
4. **Closing entry** — add a final entry to
   `skills/<slug>/_process/refactor.log` declaring the book closed.
5. **Validate** — `python3 tools/validate_skills.py`. Fix errors.

---

## 10. Cross-reference taxonomy

Use these terms in `related:` frontmatter and `## Related skills` body:

- **`extends`** — this skill builds on another and adds something new on top.
  Both still apply.
- **`refines`** — this skill is the narrower, more specific version of
  another. If both could fire, the refinement wins.
- **`contrasts`** — two skills give advice that pulls in opposite directions
  because they apply in different conditions. Each skill must name the
  *condition* under which to choose it. A contrast without a stated condition
  is a bug — sharpen the conditions or merge the skills.
- **Cross-references** — uses no taxonomy term; just describes the
  connection in prose ("Cross-references `<book>/<slug>` — the same move
  appears in...").

---

## 11. Verifying

When you think you're done:

1. **Validator** — `python3 tools/validate_skills.py`. All required
   frontmatter fields present, required body sections present, valid types,
   resolvable cross-references.
2. **Build runs clean** — `python3 docs/build.py` outputs the new
   `<slug>-skills.json` with the expected count.
3. **Local server** — `cd docs && python3 -m http.server 8765`, then visit:
   - `http://localhost:8765/` — does the new book appear in its category?
   - `http://localhost:8765/<slug>/` — does the skills page load with all
     skills, the right tag pills, and clickable related-skill links?
   - `http://localhost:8765/<slug>/log/` — does the reading log render?
4. **Hard reload** if you don't see your changes (Ctrl-Shift-R / Cmd-Shift-R).
   The `?v=` cache-busting helps but doesn't always defeat aggressive caches.

## 12. Discipline (read this every time)

- **No extended verbatim text from the source.** Frameworks, named
  techniques, chapter pointers, and brief one-line attributed quotes (when
  the exact phrasing IS the technique) are fair to cite. Long passages
  are not.
- **No invented examples.** `Worked micro-example` paraphrases a scenario
  the book itself uses. If the book has no concrete factual example to draw
  on for a given skill, omit the section rather than fabricate one.
- **The trail is committed.** Candidates, saturation, refactor logs all
  stay in `skills/<slug>/_process/`. They make the library auditable and the
  process repeatable.
- **Clarify, don't ambiguate.** Every addition must make the library more
  useful, not larger. A candidate that doesn't pass the gate does not get in.
- **Loglog is conceptual, not chronological.** Reorder for clarity; don't
  recap the chapter order.
- **Tags from the canonical set, not invented per book.** If a new tag
  truly belongs in the canonical vocabulary, add it to the FACETS list in
  both `skills-viewer.js` and `CANONICAL_TAGS.md` together — never tag a
  skill with a word that isn't in the canonical set.
